🛠️ Data Engineering & API handling

A containerized Cloud Run Job runs every day at 14:00 UTC, moving through six phases — discover_movies → discover_tv → stats → comments → excitement → predictions. Discovery is TMDB-only (zero YouTube quota); stats & comments come from the YouTube Data API; everything lands in BigQuery via idempotent MERGE so re-runs never double-write.

TMDB APISource of upcoming movies/TV & metadata. Discovery only — spends no YouTube quota.
YouTube Data API v3Daily view/like stats + top-30 comments, batched 50 ids/call (~1–2k of 10k units/day).
BigQueryServerless columnar warehouse. Writes go through staged MERGE — idempotent by design.
Cloud Run + SchedulerThe pipeline is one container, triggered on a cron. No servers to babysit.

📉 Time Series — why studios drop more trailers

A trailer’s view velocity decays geometrically — the median trailer’s half-life is ~6 days. Studios fight that decay by launching fresh trailers: 77% of launches in the data followed a measurable decline, and a new trailer injects ~2.8× the view velocity. The chart contrasts two films released around the same time — one that kept launching trailers vs. one that didn’t. ★ marks a new-trailer launch; watch the multi-trailer film re-accelerate at each star while the single-trailer film just fades.

Exponential decay / half-lifeViews fade geometrically; we fit ln(new-views) vs day to get a half-life.

🔌 MCP server — the dataset as LLM tools

A read-only Model Context Protocol server exposes the warehouse as 10 tools an LLM (e.g. Claude) can call directly — search, metrics, comments, trending, engagement & excitement decay, predictions. No SQL for the analyst; the model picks the tool. Pick one below to see its real request and response.

    MCPAn open standard that lets an LLM call your tools with typed args & structured results.
    JSON-RPC over stdioThe transport: the client spawns the process — no HTTP server, no open port.
    Read-only & least-privilegeOnly SELECTs, behind a viewer-scoped service account. It can’t mutate the warehouse.

    🧪 Model Distillation — predicting excitement decay

    Comment excitement fades as a film keeps dropping trailers — and it’s a semantic decay (fatigue, sarcasm) a keyword model can’t see. We proved it with Claude, then distilled Claude into a tiny local model so it runs daily for ~free.

    The idea, in plain terms

    A big teacher (Claude) is accurate but expensive to run on every trailer, so it labels a training set once. A tiny student learns to reproduce those labels and then runs on its own at ~zero cost. It works here because excitement lives in meaning, not keywords: frozen sentence embeddings place “this looks incredible” and “they showed the whole movie again” far apart, and a small linear head reads excitement off that geometry.

    Sentence embeddingsTurn a comment into a 384-d vector where meaning ≈ distance.
    MiniLM (all-MiniLM-L6-v2)A small, fast sentence encoder — the frozen embedder.
    Ridge regressionA 1-line L2 model maps embeddings → a 0–100 excitement score.
    ONNX / onnxruntimeRuns the encoder torch-free, so the daily job stays light.
    Order-blindingThe teacher never sees trailer order — so the student learns excitement, not position.
    GroupKFoldValidation splits by movie, so a film can’t leak across train/test.
